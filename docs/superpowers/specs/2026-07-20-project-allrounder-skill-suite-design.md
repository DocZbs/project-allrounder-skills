# Project Allrounder Skill Suite 设计

日期：2026-07-20
状态：待用户书面确认后实施

## 1. 目标

建立一个可在 GitHub 开源、可分别安装或整套安装的科研项目全生命周期 Skill 套件，服务多学科课题组的纵向项目与横向项目。

套件覆盖：机会识别、指南与需求分析、申报策划、立项交接、任务分解、研究与开发、进度汇报、经费与报销材料、技术文书与交接报告、项目 PPT、验收答辩、结项归档和经验复用。

核心原则是以项目主档案为唯一事实源，把指南或合同中的要求贯穿到任务、成果、经费、汇报和验收证据，避免不同材料之间相互矛盾。

## 2. 非目标

- 不代替项目负责人作学术判断、事实确认或最终签署。
- 不把生成式 AI 直接生成的文本冒充申请人独立完成的国自然申请书、进展报告或结题/成果报告。
- 不提供学校财务部门、法务部门或项目主管部门才能作出的正式审批结论。
- 不收录来源不明、疑似泄密、未经许可传播或版权状态不清的中标申请书和项目 PPT。
- 不承诺项目获批、通过验收或报销成功。
- 不把所有项目类别的易变规则硬编码成永久不变的结论。

## 3. Skill 套件

### 3.1 `project-allrounder`

作为唯一的全生命周期总控入口，负责：

- 识别项目类型和当前阶段。
- 建立并维护项目主档案。
- 冻结指南、合同、任务书和单位制度的版本。
- 维护需求、指标、任务、成果、证据、预算和风险之间的关系。
- 组织立项交接、研究与开发、进度控制、变更管理、报销材料、验收和结项。
- 在需要申报材料或 PPT 时调用另外两个 Skill。

### 3.2 `writing-project-proposals`

负责纵向和横向项目的申报与前期论证，覆盖：

- 当年指南、通知、申报模板和单位要求解析。
- 资格、限项、附件、伦理、保密和形式审查。
- 优秀申请案例检索与案例对照矩阵。
- 选题、问题、目标、创新、研究内容、技术路线、工作基础和指标设计。
- 横向项目需求澄清、技术方案、交付边界、知识产权、付款节点和验收条款。
- 预算逻辑与预算说明检查。
- 评审视角的反向审稿和提交前一致性审计。

对国家自然科学基金，该 Skill 采用人机协作模式：整理规则、形成问题清单、证据矩阵、提纲和审稿意见；所有事实、参考文献和最终表述由申请人核验并承担责任。

### 3.3 `building-project-presentations`

负责生成真实 PPTX，覆盖六种汇报场景：

1. 申报答辩。
2. 启动或开题汇报。
3. 中期或进度汇报。
4. 技术评审与成果演示。
5. 验收答辩。
6. 结项汇报。

该 Skill 负责项目专用的叙事、证据选择、场景路由和问答准备；实际 PPTX 创建、渲染和视觉检查复用已安装的 Presentations Skill，不复制其底层工具说明。

## 5. 项目生命周期

| 阶段 | 主要操作 | 阶段交付物 | 退出条件 |
|---|---|---|---|
| 项目接收 | 收集指南、通知、需求与制度 | 原始资料清单、来源快照 | 关键来源齐全或缺口已记录 |
| 机会评估 | 资格、匹配度、资源、周期和风险判断 | Go/No-Go 报告 | 负责人明确是否申报或承接 |
| 需求解构 | 拆解指南、评分项或合同条款 | 需求—响应矩阵 | 每条要求均有响应或缺口 |
| 案例研究 | 检索优秀申请书、评审意见和优秀 PPT | 案例对照矩阵 | 已提炼适用原则和禁用内容 |
| 申报策划 | 构建问题、目标、创新、路线和成果闭环 | 项目论证骨架 | 负责人确认核心论证 |
| 材料协作 | 撰写支持、事实核验、审校和附件准备 | 申报材料包、核验记录 | 合规、事实和格式检查通过 |
| 提交审查 | 资格、预算、附件与一致性审计 | 提交前审计报告 | 阻断项清零并人工确认 |
| 立项交接 | 把获批文本或合同转换为执行基线 | 项目主档案、任务基线 | 指标、任务和预算均入档 |
| 执行规划 | WBS、RACI、里程碑、资源和预算安排 | 执行计划 | 负责人和任务责任人确认 |
| 研究与开发 | 实验、调研、软件、样机或咨询工作 | 阶段成果与原始记录 | 成果关联任务和指标 |
| 过程控制 | 跟踪进度、风险、变更、质量和经费 | 状态报告与台账 | 偏差有责任人和处理动作 |
| 中期汇报 | 汇总计划、完成度、证据和下一步 | 进度报告、中期 PPT | 声明均有证据且人工确认 |
| 验收准备 | 指标逐项核验、缺口整改和材料编排 | 验收证据矩阵、验收包 | 每项指标有结论和证据 |
| 答辩验收 | 汇报、演示、问答和整改 | 验收 PPT、讲稿、问答库 | 验收意见已记录并落实 |
| 结项归档 | 报告、财务、成果汇交、归档和复盘 | 结项包、档案目录、复盘 | 主管要求完成且档案可检索 |

## 6. 项目主档案

项目主档案使用开放、可读、便于版本控制的文件格式。默认目录为：

```text
project-master-record/
├── 00-source/
├── 01-requirements/
├── 02-proposal/
├── 03-baseline/
├── 04-plan/
├── 05-execution/
├── 06-deliverables/
├── 07-progress/
├── 08-finance/
├── 09-changes-risks/
├── 10-acceptance/
└── 11-closeout/
```

主档案至少维护以下实体：

- 来源及版本。
- 项目要求。
- 研究问题或客户问题。
- 目标和考核指标。
- 工作包、任务、负责人和截止日期。
- 里程碑和决策。
- 成果及其版本。
- 验收证据及其文件位置。
- 预算、支出和报销凭证索引。
- 风险、问题、变更和处理记录。
- 汇报、验收和结项状态。

核心映射为：

```text
来源条款 → 项目目标 → 考核指标 → 工作包 → 任务 → 成果 → 验收证据
                              └──────→ 预算 → 支出 → 凭证
```

## 7. 项目类型规则包

第一版支持五类路由：

- 国家自然科学基金。
- 国家社会科学基金及教育部人文社科项目。
- 国家重点研发计划。
- 省市和校级纵向项目通用适配。
- 企业或机构横向项目。

通用流程保持稳定，项目类型规则按需加载。执行任务时先读取用户提供的当年指南和单位制度，再用仓库中的公开来源索引补充，不把仓库内历史规则视为最新规则。

## 8. 优秀申请案例机制

### 8.1 来源分级

- A 级：资助机构经作者许可公开的完整获批申请书和评审意见。
- B 级：中国主管部门公开的评审标准、评审综述、项目摘要、结题报告、成果报告和科技报告。
- C 级：作者或高校主动公开且能确认来源、年份、类别和获批状态的材料。
- D 级：用户有权使用的课题组历史材料，默认只在当前项目本地使用。

网络付费倒卖、来源不明、疑似内部流出或包含未经授权隐私和商业秘密的材料不进入案例库。

### 8.2 强制对照流程

正式进入申报策划前必须：

1. 冻结当前指南和评审标准。
2. 检索三至五个同类型或论证结构相近的案例；公开案例不足时明确记录不足。
3. 形成案例对照矩阵。
4. 提炼结构、证据和评审反馈规律。
5. 标出规则已过时、学科不匹配或不可迁移之处。
6. 基于当前项目真实事实形成原创论证，不复制案例表述、图表或数据。

案例对照矩阵检查：指南匹配、问题凝练、价值、创新、目标、内容、方法、基础、可行性、量化指标、风险、图表、评审优点和评审质疑。

## 9. PPT 设计

### 9.1 六类叙事

- 申报答辩：需求背景 → 核心问题 → 目标 → 创新 → 技术路线 → 基础 → 团队 → 风险与预算。
- 启动开题：任务依据 → 目标指标 → WBS → 分工 → 里程碑 → 协作机制。
- 中期进度：原计划 → 当前完成度 → 核心证据 → 偏差 → 风险 → 下一阶段。
- 技术评审：应用场景 → 方法或架构 → 核心实现 → 测试 → 演示 → 技术边界。
- 验收答辩：合同或任务指标 → 完成情况 → 逐项证据 → 成果应用 → 经费 → 验收结论。
- 结项汇报：目标回顾 → 完成度 → 代表成果 → 效益影响 → 经费 → 经验和后续计划。

### 9.2 优秀 PPT 对照

每次制作前读取至少一份合法公开或用户授权的同场景参考 PPT；没有同类材料时，使用公开科研汇报案例和已安装的演示文稿设计规则。

参考卡记录：受众、决策目标、叙事、每页任务、证据密度、技术路线表达、完成度表达、指标到证据的过渡、视觉节奏、可能追问、可借鉴原则和不可复制内容。

### 9.3 交付与门禁

必须生成真实 `.pptx`，不能只交付大纲。每份 PPT 完成：

- 内容覆盖审查。
- 事实和来源审查。
- 叙事审查。
- 指标—证据审查。
- 逐页渲染和全尺寸视觉审查。
- 文字溢出、元素重叠、裁切、换行和字体检查。
- 演讲备注、时间控制和专家问答库。
- 与任务书、报告、成果和财务数据的一致性检查。

## 10. 合规、安全和错误处理

### 10.1 信息优先级

发生冲突时按以下顺序处理：

1. 当前项目指南、专项通知和在线系统当前提纲。
2. 已签署的任务书、合同、补充协议和正式批复。
3. 当前有效的单位制度与主管部门管理办法。
4. 当前项目经负责人确认的事实和原始证据。
5. 历史优秀案例和经验性建议。

冲突不得静默覆盖，必须记录来源、差异、采用结论和确认人。

### 10.2 阻断条件

- 无法确认当前指南或模板版本。
- 关键事实、数据、引用或成果无法核验。
- 申报资格、限项、伦理、保密或知识产权存在未解决问题。
- 验收声明没有对应证据。
- 财务结论没有凭证或单位制度依据。
- 用户要求使用疑似泄密或未经授权材料。

出现阻断条件时输出缺口清单和下一步，不生成可冒充正式最终件的内容。

### 10.3 生成式 AI

国家自然科学基金 2026 年规定：借助生成式 AI 跟踪研究动态、整理参考文献时必须人工核验、对生成内容负责并按要求声明和标识；不得使用生成式 AI 直接生成申请书、进展报告或结题/成果报告，不得使用未经核实内容。

因此，国自然相关流程必须保留人工撰写、核验、声明和最终确认节点。

### 10.4 保密与隐私

- 默认只处理用户授权的数据。
- 横向项目原文、商业数据、个人信息和未公开成果不进入开源仓库。
- 公开示例必须脱敏，或使用完全虚构且明确标记为测试数据的场景。
- 生成的报告和 PPT 保留来源清单，避免错误传播。

## 11. 测试与验证

三个 Skill 逐个执行测试优先流程，完成一个再进入下一个：

1. 在未加载目标 Skill 的新上下文中运行真实场景，记录基线失败。
2. 根据失败编写最小 Skill 和必要资源。
3. 使用相同场景加载 Skill 后复测。
4. 根据新失败修订指令和资源。
5. 运行结构校验、脚本测试和触发测试。
6. 提交当前 Skill，再进入下一个。

场景至少覆盖：

- 国自然申报触发 AI 合规门禁。
- 国家社科项目正确处理申报、变更、鉴定和结项。
- 重点研发项目正确处理指南指标、任务分解、关键节点和绩效证据。
- 横向项目优先处理需求、合同、知识产权、付款和验收。
- 优秀案例对照不可跳过。
- 缺少事实时不虚构。
- 进度报告区分已完成、进行中、偏差和风险。
- 验收声明可追溯到证据。
- PPT 实际生成、逐页渲染并通过视觉检查。
- 总控 Skill 能正确调用申报和 PPT Skill。

最终集成测试使用一个完全虚构的跨学科项目，从指南接收运行到结项归档，避免使用真实敏感项目材料。

## 12. README 与开源发布

README 使用已经确认的 2×2 终端火柴人像素漫画作为主视觉，表达：

```text
老师 → 师兄 → 自己 → 项目材料堆积 → 打开 Skill 工具箱 → 有序完成
```

README 包含：一句话介绍、三 Skill 说明、安装、快速开始、端到端流程、支持类型、优秀案例机制、合规隐私、测试状态、贡献方式和许可。

图片不嵌入易出错的生成式文字，README 另提供可访问的替代文本和纯 Markdown/ASCII 流程说明。

仓库采用 MIT License。第一版完成后打 `v0.1.0` 标签；在本地验证完成前不创建远端、不推送、不发布 Release。

## 14. 验收标准

本地第一版完成时必须满足：

- 三个 Skill 均具有有效的 `SKILL.md` 和 `agents/openai.yaml`。
- 每个 Skill 可单独安装和触发。
- 总控 Skill 明确路由另外两个 Skill。
- 生命周期、项目类型、优秀案例、合规、PPT 和验收流程均有对应参考模块。
- 可复用脚本具有测试，且测试先于实现观察到失败。
- 三个 Skill 均通过官方快速校验工具。
- 每个 Skill 均完成独立前向测试。
- 集成场景从需求分析运行至结项归档。
- 示例 PPTX 经过逐页视觉检查，无未解决的高严重度问题。
- README 使用确认过的 2×2 像素漫画，安装说明可执行。
- 仓库不包含秘密、真实敏感项目材料或未经授权的申请书和 PPT。
- Git 历史由小型、可审查的提交组成。
