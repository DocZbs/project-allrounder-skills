# Horizontal-Project Route

Build a proposal that can become an executable and acceptable agreement. A persuasive technical solution is insufficient if scope, evidence, payment, or IP is ambiguous.

## Requirement Clarification

Confirm stakeholders, user problem, operating context, in-scope/out-of-scope work, inputs supplied by each party, constraints, interfaces, data access, security, deployment environment, schedule, service level, and success measures.

Record each ambiguity as a question or assumption requiring client confirmation.

## Solution and Delivery Logic

Map:

```text
client need → requirement → solution component → task → deliverable → acceptance test → evidence → payment trigger
```

For research-heavy work, distinguish exploration from guaranteed deliverables. Define how uncertainty, failed experiments, scope change, and alternative routes are handled.

## Commercial and Governance Review

Flag—not formally decide—price and tax, invoice/payment timing, reimbursable expenses, procurement, subcontracting, confidentiality, background IP, newly created IP, open-source/third-party licenses, data ownership/use/retention, publication, warranty/support, change control, delay, termination, handover, and dispute terms.

Route legal, finance, procurement, information-security, and IP conclusions to authorized owners.

## Horizontal Submission Audit

- client problem and authorized decision-makers confirmed;
- requirement baseline and exclusions explicit;
- deliverables are versioned, measurable, and owned;
- acceptance method, environment, reviewer, deadline, and remediation explicit;
- schedule and dependencies include client inputs;
- price, payment, invoice, and expense logic matches delivery;
- IP/data/confidentiality/publication terms match the technical design;
- changes require an impact assessment and authorized approval;
- handover and post-delivery obligations are bounded.
